# Release supply-chain artifacts for v1.0.0

## Status

**Proposed — recorded, not yet enforced.** This ADR captures a v1.0.0 release-pipeline architecture decision surfaced by external code review (item H6.a). It stays `proposed` until an independent multi-agent review (`ras consider`) concurs. Acceptance settles the signing/SBOM/SLSA artifact set for v1.0.0; future releases follow the same pattern unless re-opened.

## Context

`.github/workflows/release.yml` is named "Release Validation" and currently has two entry points: `push: tags: ['v*']` for real release-candidate validation, and `workflow_dispatch` for operator-triggered runs. Without explicit tag-ref guards, a branch dispatch would pass a branch name to `git verify-tag "$GITHUB_REF_NAME"` and could reach release-publishing logic from a non-tag ref. The current workflow validates the release candidate by running `go test`, `go test -race`, `govulncheck`, and `gosec`. It does **not** produce any signed, attested, or accompanying release artifact:

- No annotated/signed tag verification step (`git verify-tag` is run by consumers manually per `docs/release-verification.md`).
- No SBOM (CycloneDX or SPDX).
- No SLSA provenance.
- No cosign / Sigstore signing of source archives or built artifacts.
- No GitHub Release object created automatically.

`docs/release-checklist.md` §7 defers SBOM and signing to "if a future release adds compiled assets" — a defensible policy for the source-only `v0.x` releases but increasingly out of step with OpenSSF Best Practices and Scorecard expectations as the project approaches v1.0.0.

`docs/release-verification.md` documents the consumer-side verification flow: download the signed tag, verify with `git verify-tag` against the maintainer's SSH allowed-signers file. This is the *currently shipped* trust anchor for cpace releases. It is real — every release tag is in fact signed — but it is enforced by consumer discipline rather than by anything CI emits. A checked-in `.github/allowed_signers` file does not exist yet; this ADR requires adding it and documenting how consumers can cross-check it against `https://github.com/the-sarge.keys`.

The project is a source-only Go module: there are no compiled binaries to sign. The artifacts at release time are:

- The Git tag (annotated, signed via SSH).
- The Git tree at that tag (the source code).
- Go module proxy entries (`proxy.golang.org`), which derive deterministically from the tag.

Three classes of supply-chain artifact are candidates for v1.0.0:

**Class 1 — Tag and signature verification in CI**
- `git verify-tag` step in `release.yml` against a checked-in `.github/allowed_signers` file to be added by the implementation.
- Validates that the workflow only proceeds for annotated tags signed by the documented maintainer key.
- Locks the existing signed-tag policy into CI against maintainer error (lightweight tag, unsigned tag, or wrong signing key) and produces a public pass/fail record.
- Does **not** protect against a malicious actor who can push a `v*` tag: on a `push: tags` event, GitHub executes the workflow definition from the pushed tag's own commit, and the checked-in allowed-signers file comes from that same checkout. `Actions: write` permits workflow dispatch and re-runs, not tag creation; the primary control for tag-push authority is a repository ruleset restricting `v*` tag creation to the maintainer.

**Class 2 — Software Bill of Materials**
- SBOM generated by `anchore/sbom-action` / Syft in CycloneDX JSON with an explicit schema version.
- Lists every Go module in the dependency graph at the released tag with version and license.
- Attached to the GitHub Release as an asset, with a SHA-256 checksum published for corruption detection.
- Attested with GitHub native artifact attestations so the SBOM asset is bound to a GitHub Actions run rather than relying on a mutable release-note checksum for authenticity.
- Validates an OpenSSF Scorecard SBOM check; surfaces dependency provenance to downstream SBOM consumers (regulated environments, government procurement).

**Class 3 — SLSA Build Provenance**
- SLSA v1.0 provenance generated by `slsa-framework/slsa-github-generator`.
- Attests *which workflow run, on which commit, at which time* produced the release artifacts.
- For a source-only release, the "artifact" being attested is the Git tree at the tag plus the SBOM.
- Validates SLSA Build Level 3 with the recommended workflow.

GitHub native artifact attestations (`actions/attest-sbom`, optionally `actions/attest-build-provenance`) are a narrower mechanism than Class 3. Artifact attestations are SLSA Build Level 2 and are NOT a substitute for the deferred Level-3 `slsa-github-generator` path, but they are sufficient to authenticate the SBOM asset under GitHub's Sigstore-backed attestation flow.

The project's threat model (`docs/threat-model.md`) does not currently treat downstream consumers as the trust audience for cpace's CI infrastructure — the threat model is about the cryptographic protocol surface. Supply-chain attestations are about the *distribution* surface. Adding them is a scope expansion of what cpace warrants about how it ships.

A non-negotiable constraint from project policy: tag-signature verification in CI must not change the existing signed-tag-is-the-artifact contract for `v0.x` consumers. SBOM and SLSA are additive; they do not replace the signed Git tag.

## Decision

Adopt **Class 1 + Class 2 for v1.0.0**, plus GitHub native SBOM artifact attestation (Option F below). Defer Class 3 (SLSA Build Level 3 provenance) to a v1.x.x minor release once the SLSA generator's source-only-module support has been validated against this project.

Concretely:

- **Trigger behavior: keep both workflow entry points.** `push: tags: ['v*']` remains the only publishing path. `workflow_dispatch` remains available for rehearsals and gains a `dry_run` boolean input defaulting to `true`; dispatch runs must be invoked against a signed tag ref to exercise the pipeline. The `verify-tag` and `release` jobs require `github.ref_type == 'tag'`, and actual publishing is gated on `github.event_name == 'push' && startsWith(github.ref, 'refs/tags/v')`.
- **Class 1: Add `git verify-tag` to `release.yml`.** The workflow checks out the tag, fetches the remote tag object deterministically, asserts `git cat-file -t "$GITHUB_REF_NAME"` returns `tag`, then verifies the tag's SSH signature against `.github/allowed_signers`. The implementation adds that file and updates `docs/release-verification.md` to reference it, with a note that consumers should cross-check the checked-in copy against `https://github.com/the-sarge.keys`. The workflow fails before any other validation if `git verify-tag` fails because every other job declares `needs: verify-tag` directly or transitively. This locks the existing signed-tag policy into CI against maintainer error rather than leaving it consumer-side-only. It does not protect against a malicious tag pusher; the required repository ruleset restricting `v*` tag creation to the maintainer is the primary control for that threat. Fetching signers from `origin/main` may be added only as defense in depth, not as the primary mitigation, because the workflow definition itself still comes from the tag.
- **Class 2: Generate, attach, and attest an SBOM.** Add a SHA-pinned `anchore/sbom-action` step generating a CycloneDX JSON SBOM with an explicit schema version, explicit output path, and pinned Syft version. Upload as a workflow artifact, attach to the GitHub Release object on tag pushes that pass all gates, append a checksum section to the release body for corruption detection, and attest the SBOM with a SHA-pinned `actions/attest-sbom` step.
- **Class 3: Defer.** SLSA Build Level 3 for source-only Go modules remains outside v1.0.0. GitHub artifact attestations are SLSA Build Level 2 and are not a substitute for the deferred Level-3 `slsa-github-generator` path. Re-open this ADR when the Level-3 source-only-module flow is stable, or when a downstream consumer explicitly requests it.

The workflow also gains a GitHub Release publishing step implemented with the `gh` CLI, not a new release action. It publishes the release notes from `CHANGELOG.md` for the tagged version, appends a CI-generated checksum section, attaches the SBOM and checksum assets, and links to `docs/release-verification.md`.

## Acceptance criteria

The implementation must satisfy these before this ADR moves `proposed → accepted` *and* before v1.0.0 is tagged:

- **Workflow triggers and guards** keep `push: tags: ['v*']` and `workflow_dispatch`. `workflow_dispatch` defines a `dry_run` boolean input defaulting to `true`. The `verify-tag` and `release` jobs are guarded with `github.ref_type == 'tag'`; real publishing is additionally guarded with `github.event_name == 'push' && startsWith(github.ref, 'refs/tags/v')`.
- **Tag authority control** exists before v1.0.0: a GitHub repository ruleset restricts `v*` tag creation to the maintainer.
- **`.github/allowed_signers` is created** with the documented maintainer principal mapping. `docs/release-verification.md` references that path and tells consumers to cross-check the checked-in copy against `https://github.com/the-sarge.keys`, because the checked-in file shares the tag's trust domain.
- **`git verify-tag` runs first** in `release.yml` on any tag ref and all other jobs declare `needs: verify-tag` directly or transitively. A `verify-tag` failure prevents every other job from starting. Workflow fails if the tag is not annotated or not signed by an allowed-signer in `.github/allowed_signers`; lightweight tags trigger the workflow but immediately fail this step. The implemented workflow passes `actionlint`.
- **Deterministic tag-object verification** is implemented in `verify-tag`: after checkout, run `git fetch --force origin "refs/tags/$GITHUB_REF_NAME:refs/tags/$GITHUB_REF_NAME"`, assert `git cat-file -t "$GITHUB_REF_NAME"` prints `tag`, then run `git verify-tag "$GITHUB_REF_NAME"`. The rehearsal run records `git cat-file` printing `tag` on a signed annotated tag.
- **SBOM generation** via the real, loadable `uses: anchore/sbom-action@<sha-pin>` target produces `cpace-${{ github.ref_name }}.cdx.json` with `format: cyclonedx-json@1.5`, an explicit `output-file`, and a pinned `syft-version`. CI validates the generated BOM's `bomFormat`, `specVersion`, and presence of expected Go module entries before any release upload.
- **SBOM attestation** uses SHA-pinned `actions/attest-sbom@<sha-pin>` on the generated SBOM, satisfying the `docs/release-verification.md` release-asset policy that each asset must be signed directly or included in a signed manifest.
- **GitHub Release object** is created automatically for the tag only on publishing-eligible tag pushes, with:
  - Title: the tag name.
  - Body: the relevant `CHANGELOG.md` section for that version plus a CI-appended checksum section.
  - Assets: the SBOM file plus its SHA-256 checksum.
- **Release-note extraction is fail-closed**: if the extracted `CHANGELOG.md` section for the tag is absent or empty, the extraction step exits non-zero and blocks publishing.
- **`docs/release-verification.md` is updated** to describe SBOM checksum comparison as corruption detection only, and to document optional layered SBOM attestation verification with `gh attestation verify`.
- **`docs/security-gates.md` and `docs/release-checklist.md` are updated** to list SBOM as a release deliverable for v1.x.
- **`docs/threat-model.md` or `docs/ci-policy.md` gains a distribution-surface section** covering tag authority, what CI does and does not attest, and the SBOM authenticity posture under the adopted attestation.
- **No SLSA Build Level 3 provenance** is generated. The ADR explicitly defers it; the next-step entry in `docs/release-checklist.md` says so.
- **Action SHA pinning** for `anchore/sbom-action`, `actions/attest-sbom`, and any other new actions, matching the existing project convention.
- **Workflow permissions** preserve least privilege: top-level permissions remain `contents: read`; only the release-publishing job gets `contents: write` for `gh release create` / `gh release edit` / asset upload; only the SBOM-attestation job gets `id-token: write` and `attestations: write` for `actions/attest-sbom`.
- **Rehearsal and negative tests** are recorded before v1.0.0: a guarded `workflow_dispatch` dry run against a signed tag ref runs `verify-tag`, SBOM generation, and SBOM attestation without creating or mutating a GitHub Release; fork-based negative tests show a lightweight tag, unsigned tag, and non-allowed-signer tag each fail `verify-tag` with no downstream jobs starting; a tag absent from `CHANGELOG.md` causes the release job to fail before publishing.
- **Scorecard evidence** is recorded before and after implementation with `--checks=SBOM,Packaging,Signed-Releases`, matching the project's evidence-discipline policy and the README's visible Scorecard badge.

## Considered options

- **A — Class 1 + Class 2, defer SLSA.** Verifies tag signatures in CI and provides SBOM to downstream consumers. Does not commit to SLSA Level-3 tooling that may need rework.

- **B — Class 1 + Class 2 + Class 3 (full OpenSSF posture).** All three artifact classes from the start. Larger CI surface, more dependency on the SLSA generator's quirks. If a SLSA-generator regression breaks releases, recovery requires temporary disabling.

- **C — Class 1 only.** Adds tag-signature verification but keeps the source-only-no-extra-artifacts policy. Simplest, no SBOM for downstream consumers.

- **D — Status quo.** Defer all of Class 1, 2, and 3 to a later release. Continue to rely on consumer-side `git verify-tag` discipline.

- **E — Cosign artifact signing.** Add `cosign sign` over a source-archive tarball plus the SBOM. Establishes a Sigstore-anchored trust path for the assets. Has the same dependency-on-tooling-maturity question as SLSA; not all downstream consumers currently verify Sigstore signatures for Go modules.

- **F — GitHub native artifact attestations (adopted for the SBOM).** Add `actions/attest-sbom` over the generated SBOM. Establishes a GitHub/Sigstore-backed authenticity path for the SBOM without adopting the full SLSA Level-3 generator. Consumer verification still carries friction (`gh` CLI plus Sigstore trust), so it is documented as optional and layered.

## Consequences

- **Option A + F (adopted):**
  - One CI run produces a signed-tag verification artifact (pass/fail) plus an SBOM.
  - GitHub Release object becomes the canonical "what shipped" pointer.
  - New action dependencies are `anchore/sbom-action` and `actions/attest-sbom`, both pinnable by SHA. Release creation uses the existing `gh` CLI, so there is no new release-action dependency.
  - Downstream consumers gain SBOM visibility. The release-notes checksum is convenience-only corruption detection because the release asset and checksum share the mutable release trust domain.
  - SBOM authenticity comes from the GitHub artifact attestation, verifiable with `gh attestation verify`; that verification is optional and layered. The primary consumer trust path remains the Class-1 signed tag.
  - SLSA Build Level 3 can be added in a v1.x.x minor release as additive metadata.
  - Tag-push-triggered validation is reactive with respect to Go module proxy/sumdb immutability; the pre-tag local gate in `docs/release-checklist.md` remains the prevention layer, and a failed run means cutting a superseding tag, not retracting the bad one.
  - Residual risk: a principal who can push a `v*` tag controls the tag's workflow definition and checked-in signer list, so CI tag verification does not protect against malicious tag pushers. The repository ruleset restricting `v*` tag creation is the primary control; sourcing signers from `origin/main` would be defense in depth only.
  - OpenSSF Scorecard impact must be measured with before/after evidence:

    | Check | Before | After Option A + F | Evidence requirement |
    | --- | --- | --- | --- |
    | SBOM | No release SBOM is generated. | Improves because the release includes a generated SBOM. | Record before/after `--checks=SBOM,Packaging,Signed-Releases`. |
    | Packaging | Source-only Go module with no CI-published package signal. | Likely remains inconclusive for a source-only Go module's GitHub Release. | Record before/after and do not claim a Packaging gain unless Scorecard shows one. |
    | Signed-Releases | No attached release assets, so the check may be inconclusive rather than failing. | Unsigned release assets would regress; the adopted SBOM attestation resolves that regression for the SBOM asset. | Record before/after and retain attestation evidence. |

- **Option B:**
  - All three artifact classes from v1.0.0. Strongest supply-chain posture.
  - Highest dependency on third-party tooling. If `slsa-framework/slsa-github-generator` has a regression for source-only modules, the release pipeline breaks.
  - Mitigation: gate SLSA generation behind a workflow input flag so it can be disabled per-release.

- **Option C:**
  - Verifies tag signature in CI; that alone closes the highest-leverage hardening from the review (item H6.a minimum).
  - No SBOM for downstream consumers; Scorecard SBOM check stays failing.
  - Smaller CI surface to maintain.

- **Option D:**
  - Zero CI change. Signed-tag verification remains consumer-side.
  - Loses ground against OpenSSF Best Practices expectations for v1.0.0.

- **Option E:**
  - Sigstore signatures over source archive + SBOM.
  - Trust anchored in Sigstore root rather than the maintainer's SSH key.
  - Requires consumers to install `cosign` and trust Sigstore's root; not all downstream Go consumers do.
  - Useful complement to options A/B/C, not a replacement.

- **Option F:**
  - GitHub artifact attestations authenticate the SBOM asset and resolve the Signed-Releases regression introduced by attaching release assets.
  - Artifact attestations are SLSA Build Level 2 and are NOT a substitute for the deferred Level-3 `slsa-github-generator` path.
  - Consumer verification requires `gh` CLI support and Sigstore trust, similar in kind to the cosign objection, so it remains optional/layered rather than the primary consumer trust path.

## Implementation outline (Option A + F)

1. Add `.github/allowed_signers` containing the documented maintainer principal mapping, then add a `verify-tag` job to `.github/workflows/release.yml` that:
   - Runs only when `github.ref_type == 'tag'`.
   - Has no `needs` dependencies.
   - Checks out with `fetch-depth: 0` using the project's pinned `actions/checkout` version.
   - Runs `git fetch --force origin "refs/tags/$GITHUB_REF_NAME:refs/tags/$GITHUB_REF_NAME"`.
   - Asserts `git cat-file -t "$GITHUB_REF_NAME"` returns `tag`.
   - Runs `git config gpg.ssh.allowedSignersFile .github/allowed_signers` and `git verify-tag "$GITHUB_REF_NAME"`.
   - Fails the workflow if any of those commands exits non-zero.
   - Defines the fail-fast needs graph: `verify-tag` → `check` / `race` / `vuln` / `gosec` → `sbom` → `attest-sbom` → `release`; every job after `verify-tag` depends on it directly or transitively.
2. Update `workflow_dispatch` to include `dry_run` as a boolean input defaulting to `true`. Dispatch rehearsals must select a signed tag ref; branch dispatches do not run tag verification or publishing because `verify-tag` and `release` require `github.ref_type == 'tag'`.
3. Keep top-level workflow permissions at `contents: read`. Grant `contents: write` only on the `release` job; grant `id-token: write` and `attestations: write` only on the `attest-sbom` job.
4. Add an `sbom` job that:
   - Uses `anchore/sbom-action@<sha-pin>`.
   - Sets `format: cyclonedx-json@1.5`.
   - Sets an explicit `output-file`, for example `cpace-${{ github.ref_name }}.cdx.json`.
   - Pins `syft-version`.
   - Sets `upload-release-assets: false` so release publishing remains in the `release` job.
   - Uploads the SBOM as a workflow artifact.
5. Add an SBOM validation step that asserts `bomFormat == "CycloneDX"`, `specVersion == "1.5"`, and expected Go module entries are present; then compute `sha256sum` for the SBOM.
6. Add an `attest-sbom` job that uses `actions/attest-sbom@<sha-pin>` on the generated SBOM artifact.
7. Add a `release` job that:
   - Runs after all gates pass and only when `github.ref_type == 'tag'`.
   - Publishes only when `github.event_name == 'push' && startsWith(github.ref, 'refs/tags/v')`; `workflow_dispatch` dry runs exercise extraction and artifact preparation without creating or mutating a GitHub Release.
   - Uses the `gh` CLI with an idempotent create-or-edit flow such as `gh release view "$tag" >/dev/null 2>&1 && gh release edit "$tag" ... || gh release create "$tag" ...`, with asset upload using `--clobber` or the equivalent guarded replacement behavior.
   - Extracts the relevant `CHANGELOG.md` section for the version with a small script that exits non-zero when the section is absent or empty.
   - Appends one checksum section generated by CI to the release body.
   - Attaches the SBOM and its SHA-256 checksum.
8. Update `docs/release-verification.md` to add SBOM corruption checking by SHA-256, optional `gh attestation verify` for SBOM authenticity, the `.github/allowed_signers` path, and the consumer cross-check against `https://github.com/the-sarge.keys`.
9. Update `docs/release-checklist.md` §7 to remove the "if a future release adds compiled assets" qualifier and replace with the new policy, including the pre-tag local gate remaining the prevention layer for Go proxy/sumdb immutability.
10. Update the `docs/security-gates.md` Release Validation description to list SBOM and SBOM attestation among gates.
11. Update `docs/threat-model.md` or `docs/ci-policy.md` with a short distribution-surface section covering tag authority, what CI does and does not attest, and the SBOM authenticity posture.
12. Add a `CHANGELOG.md` Unreleased entry recording the new release artifacts.
13. Record before/after Scorecard evidence with `--checks=SBOM,Packaging,Signed-Releases`.
14. Rehearse before tagging v1.0.0:
   - Run a guarded `workflow_dispatch` dry run against a signed tag ref and confirm it runs `verify-tag`, SBOM generation, and SBOM attestation without publishing; the log must show `git cat-file -t "$GITHUB_REF_NAME"` printing `tag`.
   - In a fork, push a lightweight tag, unsigned tag, and non-allowed-signer tag; each must fail `verify-tag` with no downstream jobs starting.
   - In a fork, push a signed tag absent from `CHANGELOG.md`; the release job must fail before publishing.
